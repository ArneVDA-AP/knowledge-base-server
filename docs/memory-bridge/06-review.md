# Memory-Bridge Review

## CONFIRMED

- [HIGH] src/db.js:971-976 (importMemories) — `kb_remember` MCP tool accepts client-supplied `created_by`, letting an authenticated agent forge `user` provenance over the network-exposed MCP HTTP transport. `created_by==='user'` auto-sets `review_status='accepted'`/`confidence='asserted'`/`source='memory:user'` (db.js:659-669), bypassing the propose/dispose review queue. REST routes hardcode provenance correctly (v1.js:380 'agent', api.js:200 'user'); only the MCP tool leaks it. — FIX: drop the `created_by` zod field from kb_remember (tools.js:469) and hardcode `created_by:'agent'` in the handler (tools.js:475), matching v1.js. If a local user-write is ever needed, gate it behind a separate admin-only tool.

- [HIGH] src/db.js:971-976 (importMemories) — Export/import round-trip silently drops `outcome_score`, the learned reward signal. `exportMemories` serializes it (MEMORY_EXPORT_FIELDS, db.js:944) but `importMemories` rebuilds rows via `insertDocument()`, whose param/INSERT lists (db.js:186-188, 207-211) omit it, so every import resets it to DEFAULT 0. Verified empirically (export=1 → re-import=0). Wipes the `recordMemoryOutcome` helped/burned trust history while the export file misleadingly still contains it; the "provenance preserved" comment overclaims. — FIX: after `insertDocument` returns in importMemories, run `db.prepare('UPDATE documents SET outcome_score = ? WHERE id = ?').run(m.outcome_score ?? 0, doc.id)` (import skips duplicates by content_hash, so the returned row is always fresh). Decide separately whether to also export/preserve `predicted_outcome`.

- [HIGH] src/consolidate.js:91 — `consolidate()` crashes when the extractor LLM emits a candidate with non-string `content`. The guard at consolidate.js:76 checks truthiness only; an object/number `content` reaches the un-try/caught `rememberMemory` call, where `Buffer.byteLength(content)` (db.js:672) throws a TypeError, aborting the whole loop and losing every not-yet-written candidate (CLI: uncaught stack; MCP kb_consolidate: isError). Reproduced end-to-end (0 documents written). LLM output is untrusted. — FIX: tighten the guard to `if (!c || typeof c.title !== 'string' || typeof c.content !== 'string') { result.skipped++; continue; }`, and wrap the write block (consolidate.js:91-108) in try/catch counting failures as skipped. Best also coerce/validate `content` at the keystone `insertDocument`.

- [MEDIUM] src/db.js:969 (importMemories) — `importMemories` crashes on one NDJSON line with non-string `content`, aborting the whole import. Line 968's `if (!m||!m.title||!m.content)` is truthiness-only, so `{"title":"x","content":42}` reaches `createHash('sha256').update(m.content)` (db.js:969) and throws TypeError; the throw is uncaught, so every valid line after the bad one is silently lost and the CLI prints a raw stack instead of "skipped". NDJSON is the documented untrusted portability format. Reproduced: 3-line file (good/bad/good) imports 0 and throws. (Downgraded from high: reachable only via hand-edited/corrupt files through the local CLI, no network surface.) — FIX: change the check to `if (!m || typeof m.title !== 'string' || typeof m.content !== 'string') { skipped++; continue; }` and/or wrap the per-line body (969-978) in its own try/catch doing `skipped++; continue;`. Coerce at insertDocument too.

- [MEDIUM] src/db.js:971-975 (importMemories) — `importMemories` trusts `created_by`/`review_status`/`confidence` straight from the untrusted NDJSON file; `insertDocument` does no enum validation, so a crafted line lands a `created_by:'user'`, `review_status:'accepted'`, `confidence:'verified'` memory directly into getSessionBrief CORE (db.js:895-898) with full trust, never passing human review. Bypasses the central provenance invariant (rememberMemory derives these server-side, db.js:660-661). Local/supply-chain vector (CLI import only; no REST route exposes import). — FIX: on import, preserve `created_by` (the round-trip test depends on it) but force `review_status` to 'pending' (or 'imported') and cap `confidence` at 'inferred' for unvouched files, unless an explicit `--trust` flag is passed. Update the round-trip test accordingly. Document that import re-enters the review queue.

- [MEDIUM] src/db.js:971-976 (importMemories) — Import discards the original `created_at`, corrupting recency/salience of an imported brain. `created_at` is in MEMORY_EXPORT_FIELDS (db.js:944) but `insertDocument` doesn't accept it, so every imported row gets a fresh CURRENT_TIMESTAMP; since `last_accessed_at` is also null, `retrievability()` anchors on `created_at` (db.js:475) and every imported memory reads as maximally recent, artificially boosting it in recall ranking. Verified: a 2020-backdated memory re-imports with today's timestamp. — FIX: after insertDocument in importMemories, `if (m.created_at) db.prepare('UPDATE documents SET created_at = ? WHERE id = ?').run(m.created_at, doc.id)` (or add created_at as an optional insertDocument param). If not preserving, drop it from MEMORY_EXPORT_FIELDS.

- [MEDIUM] tests/memory.test.js:317-322 — The "T=0 recall is deterministic" test calls `recallMemories('ndet', {limit:4})` with NO `temperature` argument, so it depends on `ENV_RECALL_TEMP` (read once at module load from `KB_RECALL_TEMPERATURE`, db.js:503/751). With 6 candidates > limit 4 and any temp>0, it takes the Date.now()-seeded Gumbel path and flakes across millisecond boundaries; today it passes only because the env var happens to be unset. The project's #1 backward-compat invariant ("T=0 reproduces exact top-k") is thus not actually locked. (Downgraded from high: no production impact; wrong-passing/flaky only if the env var is set.) — FIX: pass `temperature: 0` explicitly; additionally assert the order equals a pure salience sort and is seed-independent.

- [MEDIUM] tests/memory.test.js:348-356 — The replay-queue test claims "ranks by importance × surprise, not FIFO" but the surprise term is not load-bearing: `replay_priority = importance × (1+|outcome_score|)` (db.js:936) already orders hi(0.9) before lo(0.2) with zero surprise, so an implementation ignoring `outcome_score` entirely would pass. "Not FIFO" is trivially satisfied since the higher-importance item was inserted second. (Downgraded from high: test-discrimination gap, getReplayQueue itself is correct.) — FIX: give both memories equal importance (e.g. 0.5), burn only one, and assert the burned one ranks first; optionally add product-ordering cases.

- [MEDIUM] tests/memory.test.js:324-330 — The temperature/seed test has no negative control: it asserts only reproducibility (r1===r2) and 3 distinct ids, both of which a degenerate `scored.slice(0,limit)` (ignoring temperature) would also satisfy. The core Gumbel-top-k stochasticity claim is unverified, worsened by all 8 fixtures using importance 0.5 (near-degenerate order). — FIX: add a negative control — with a real salience spread, assert a high-temperature sample differs from the temperature:0 top-k in at least one position, and that two different seeds at T=1.0 produce different samples.

- [LOW] src/tools.js:476 / src/routes/v1.js:374,380 → src/db.js:661 — Remote agents can self-declare `confidence:'verified'` on writes; the client value takes precedence over the `'inferred'` agent default (db.js:661), with no server-side clamp, contradicting the "agent reasoning enters at inferred, not promoted to verified without a ground-truth check" guardrail (03 §7.4). - [LOW] src/db.js:198 & 677 (insertDocument/rememberMemory) — `importance` is never clamped to its documented [0,1] range on any write path (only null-coalesced to 0.5). `salienceOf` uses it linearly and getSessionBrief CORE orders by `importance DESC`, so a finite `importance: 999` permanently dominates recall and pins CORE; negative values suppress. Not a DoS (z.number rejects NaN, JSON can't carry Infinity). — FIX: clamp on write — `importance: Math.min(1, Math.max(0, importance == null ? 0.5 : Number(importance)))` in insertDocument, and apply the same in importMemories.
